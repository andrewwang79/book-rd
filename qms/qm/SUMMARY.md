# 质量管理

## 管理
| 比对项 | 配置控制委员会（CCB） | 变更控制委员会（CCB） |
| - | - | - |
| 对应管理 | 配置管理(CM)：**状态管控的“基础载体”，“状态可查”** | 变更管理(MOC，Management of Change)：**状态变更的“管控流程”，“变更可控”** |
| 流程 | 基线建立→配置项变更→CMDB 更新→配置审计 | CR发起→评审→CO下达→CA执行→变更关闭 |
| 核心目标 | 管控配置基线：完整性、一致性、可追溯性 <br/> **“配置是否准确”：确保配置项与基线一致，CMDB记录与实物匹配** | 管控所有可能影响配置基线的变更：必要性、可行性、合规性 <br/> **“变更是否该做”：确保变更有价值、风险可控、符合法规要求** |
| 管理对象 | 配置项（如代码、BOM、文档、版本号）和配置基线 | CR、CO、CA |
| 核心职责 | 1. 审批配置项的新增、修改、删除 <br/> 2. 批准配置基线的建立、更新、冻结 <br/> 3. 裁决配置管理中的冲突（如基线版本不一致） <br/> 4. 审核 CMDB（配置管理数据库）的更新内容 | 1. 评审CR的必要性、技术可行性、临床安全性、合规风险 <br/> 2. 批准/否决/暂缓变更，下达变更指令（CO） <br/> 3. 指定变更负责人（CRP），监督变更执行 <br/> 4. 审批变更终止 |

## 变更管理
### CCB子委员会
| 角色 | 核心定位 | 核心职责 | 工作项 |
| - | - | - | - |
| CRB(Change Review Board)变更评审委员会 | 变更的“业务决策机构” | 业务、技术、成本 | 1. 评估变更的业务必要性(业务价值)、技术可行性(技术方案，技术风险，兼容性)、成本合理性 <br/> 2. 向CCB提交《变更评审报告》（专业评审意见与决策建议） <br/> 3. 制定变更实施的资源计划与风险控制预案，跟踪变更实施进度，向CCB同步风险动态 |
| CRB(Clinical Review Board)临床评审委员会 | 医疗变更的“临床安全决策机构” | 临床安全，有效性评审 | 1. 评估变更对患者安全、诊断/治疗效果的影响；审核变更的临床适用性、伦理合规性；验证变更的临床效益大于风险 <br/> 2. 向CCB提交《变更评审报告》（专业评审意见与决策建议） <br/> 3. 制定变更实施的资源计划与风险控制预案，跟踪变更后临床使用反馈，向CCB同步风险动态 |

### 公司岗位设置
* 大企业：2个CCB + CCB的子委员会
    1. 统一决策出口，避免多头管理：专项委员会的评审意见最终都汇总到CCB，由CCB给出唯一的审批结论，解决了“技术部门说可行、临床部门说有风险”的多头决策问题。
    1. 专业化分工，降低决策风险：CCB无需深入每个专业领域的细节，而是依靠下属的专项委员会提供专业意见，既避免了「非专业决策」的漏洞，又能聚焦于全局统筹。
* 中小企业或单一产品线：2个CCB合并为一个“变更与配置控制委员会”，没有CCB的子委员会
