# 质量管理

## 流程
| 流程 | 核心定位 | 核心职责 | 管理对象范围 |
| - | - | - | - |
| **配置管理CM)** | 状态管控的“基础载体”，“状态可查” | 建立配置基线、维护CMDB(配置管理数据库)、记录配置项全生命周期状态、开展配置审计 | 医疗设备/软件的硬件、软件、文档、版本等所有配置项 |
| **变更管理(MOC，Management of Change)** | 状态变更的“管控流程”，“变更可控” | 规范变更申请-评估-审批-实施-验证的全流程，核心是**风险管控与合规性把关** | 所有可能影响配置基线的变更行为（硬件升级、软件迭代、流程调整等） |

## CCB
| 角色 | 核心定位 | 核心职责 | 工作项 |
| - | - | - | - |
| **CCB(配置控制委员会)** | 变更与配置的最终决策中枢 | **业务技术可行、临床安全、流程合规** | 1. 对变更的**技术可行性、临床安全性、配置一致性、版本兼容性**等进行评审 <br/> 2. 管控配置基线的建立、冻结、更新与解冻 <br/> 3. 裁决跨专项委员会的评审争议 <br/> 4. 监督变更实施的合规性与配置一致性 <br/> 5. 对变更全流程的风险与合规性负责 |
| **CRB(变更评审委员会，Change Review Board)** | 变更的“业务决策机构”，CCB下属 | **业务、技术、成本** | 1. 评估变更的业务必要性(业务价值)、技术可行性(技术方案，技术风险，兼容性)、成本合理性 <br/> 2. 向CCB提交《变更评审报告》（专业评审意见与决策建议） <br/> 3. 制定变更实施的资源计划与风险控制预案，跟踪变更实施进度，向CCB同步风险动态 |
| **CRB(临床评审委员会，Clinical Review Board)** | 医疗变更的“临床安全决策机构”，CCB下属 | **临床安全，有效性评审** | 1. 评估变更对患者安全、诊断/治疗效果的影响；审核变更的临床适用性、伦理合规性；验证变更的临床效益大于风险 <br/> 2. 向CCB提交《变更评审报告》（专业评审意见与决策建议） <br/> 3. 制定变更实施的资源计划与风险控制预案，跟踪变更后临床使用反馈，向CCB同步风险动态 |

* 大公司：会有这3个角色
  1. **统一决策出口，避免多头管理**：专项委员会的评审意见最终都汇总到CCB，由CCB给出唯一的审批结论，解决了“技术部门说可行、临床部门说有风险”的多头决策问题。
  1. **专业化分工，降低决策风险**：CCB无需深入每个专业领域的细节，而是依靠下属的专项委员会提供专业意见，既避免了「非专业决策」的漏洞，又能聚焦于全局统筹。
* 小公司：只有CCB，CCB承担2个CRB的工作
