# 变更管理
## 流程
* 变更申请人和负责人是不同的角色。需求类变更的申请人是产品经理。
* 变更负责人的职责：流程管理者，变更的合规性，风险和执行闭环
* 4和5就是研发流程

| 一级 | 二级 | 说明 |
| - | - | - |
| **1. 变更发起** | 1.1. 触发变更的场景识别 | 申请人识别变更场景，如硬件升级、软件版本迭代、设备使用流程、校准参数的修改 |
| | 1.2. 提交《变更申请单》 | 见"变更申请材料" |
| **2. 变更分类与风险评估** | 2.1. 变更分级 | 由研发/质控/临床部门评估等级 |
| | 2.2. 风险评估 | 组建技术+临床+合规评估小组，评估变更影响和风险 |
| **3. 变更评审与审批** | 3.1. 变更评审与审批 | 基于变更等级CCB审批 |
| **4. 变更实施(受控执行)** | 4.1 制定实施计划 | 明确实施时间、责任人、操作步骤、应急预案；<br>涉及临床使用的变更，提前培训医护人员 |
| | 4.2. 受控执行 | 技术类变更先在测试环境验证，再部署生产环境；<br>记录实施全过程（操作时间、参数修改、参与人员），形成《变更实施记录表》 |
| | 4.3. 应急处置 | 实施中出现异常（如软件卡顿、设备性能下降），立即暂停变更，启动应急预案，恢复至变更前配置基线状态 |
| **5. 变更验证与确认** | 5.1. 技术验证 | 技术人员测试系统性能、功能是否符合预期，核对配置项状态与计划一致性 |
| | 5.2. 临床确认（中度/重大变更） | 由CRB或临床科室开展小范围试用，收集反馈；<br>验证指标：诊断准确率、治疗有效率、操作便捷性等，形成《临床验证报告》 |
| | 5.3. 合规检查 | 确认变更后产品/系统符合注册证要求、行业标准 |
| **6. 变更关闭与记录归档** | 6.1. 变更关闭 | 验证通过：审批人签署《变更关闭确认书》，结束流程；<br>验证不通过：退回至“变更申请”环节，重新优化方案 |
| | 6.2. 配置记录更新 | 同步更新至CMDB（配置管理数据库）：<br>• 配置项版本号、参数值；<br>• 变更历史（原因、时间、责任人）；<br>• 重大变更需冻结新配置基线 |
| | 6.3. 文档归档 | 整理归档所有材料：《变更申请表》《风险评估报告》《评审意见》《实施记录》《验证报告》等；<br>保存期限：至少至产品生命周期结束后5年 |
| **7. 变更后监控与审计** | 7.1. 日常监控 | 变更后1-3个月内，跟踪产品/系统运行状态，收集临床反馈，及时处理潜在问题 |
| | 7.2. 配置审计 | 定期核查实际配置与CMDB记录一致性，确保无非授权变更 |

* 场景示例：某手术导航软件 V2.0→V2.1 升级（优化病灶定位算法，属于临床相关变更）：
  1. 配置管理：当前基线为软件 V2.0 版本，所有相关代码、文档、临床验证报告均记录在CMDB中
  1. 变更申请：研发团队提交变更申请，说明算法优化内容及预期临床效益
  1. CRB 评审：评估算法变更对手术定位精度的影响，要求开展小范围临床试用验证
  1. CCB 评审：评估软件升级的技术可行性、与硬件的兼容性、版本控制方案
  1. 变更实施：根据双评审意见，在测试环境完成升级，开展临床试用
  1. 验证确认：CRB 确认临床试用效果达标，CCB 确认技术指标符合要求
  1. 配置更新：CMDB中将软件版本更新为 V2.1，冻结新基线
  1. 配置审计：定期核查临床使用的软件版本是否与基线一致，CCB裁决审计偏差

### 变更申请材料
| 材料分类 | 材料内容 |
| - | - |
| 变更基本信息 | 变更描述、变更对象（设备/软件型号、版本）、变更原因、变更内容、预期目标 |
| 变更影响 | 1. 影响和范围 <br> 2. 影响评估：产品质量（性能、可靠性和安全性），生产流程，客户满意度，法规合规性 |
| 附加信息 | 技术方案、临床需求说明等支撑材料 |

## 资料
### DFMEA
> 设计的FMEA（失效模式与效应分析，Failure Mode and Effects Analysis）

* 目的：设计阶段用来评估和优化设计的工具。识别设计中可能存在的风险、问题或不足之处，评估它们对产品性能和可靠性的可能影响，并提出改进措施。通过DFMEA团队可以提前识别并解决这些潜在问题，从而提高产品的可靠性和安全性。系统地开发出既符合需求，又可靠安全的产品。
    1. 风险评估：识别，分析，评估，控制
    1. 风险控制措施
* 需DFMEA的需求
    1. 关键功能需求：优先分析对产品性能、安全性能或可靠性至关重要的功能需求。
    1. 高风险功能需求：可能导致严重后果或风险的功能需求。
    1. 法规和标准要求：符合法规和标准的功能需求
* 评估的风险点
    1. 患者安全风险：是否导致误诊、漏诊、治疗失误
    1. 技术风险：是否破坏系统兼容性、稳定性
    1. 合规风险：是否需向NMPA/FDA等监管部门报备或重新注册

### 变更等级
| 变更等级 | 判定标准 | 举例 |
| - | - | - |
| 微小 | 不影响产品安全性、有效性，无临床风险 | 软件界面文字修改、设备常规保养参数微调、文档格式优化 |
| 中度 | 对产品性能有一定影响，但风险可控，不改变核心功能 | 软件非核心功能优化、设备次要部件更换、常规参数调整 |
| 重大 | 影响核心功能、临床诊断 / 治疗效果，或需满足监管报备要求 | 医疗器械软件核心算法修改、设备关键性能参数调整、诊断标准变更 |

### 变更评审流程
| 变更等级 | 评审人员 | 流程 |
| - | - | - |
| 微小 | 研发 / 质量 | 无需上会，直接审核《变更申请表》，确认无风险后批准 |
| 中度 | CCB（技术评审） | CCB |
| 重大 | CCB（临床安全评审 + 技术评审） | CCB |

### CCB评估项
1. 项目延迟：可能导致项目的完成时间比原计划延迟很多
1. 超出预算：可能导致项目的成本超出预算
1. 质量下降：可能导致产品或服务的质量显著下降
1. 满意度下降：可能导致客户或用户的满意度显著下降
1. 合规风险：可能导致组织违反法规或标准
